\projecttitle{SQL Query Optimizer and Execution Engine}{C or C++}

\section{Problem Statement}

You will build a small SQL query engine that takes a SQL string, turns it into a query plan, applies one classical optimization to that plan, and then executes it against tables stored on disk. The result is an interactive shell where a user types a SQL query and gets back rows. The same shell can show the user the plan for a query both before and after optimization, side by side, so the speedup is visible.

You are not building a full database. You are building the part of a database that sits between the SQL the user types and the rows the user gets back. The storage layer is intentionally trivial: tables live as CSV files on disk and are read sequentially. There are no indexes, no transactions, no concurrency. Your effort goes into the four things this project is actually about: parsing a small subset of SQL, representing a query as a tree of operators, transforming that tree to make it faster, and executing the transformed tree row by row.

The single optimization you will implement is called \textit{predicate pushdown}. It is the most important optimization in any SQL engine, and your benchmark will show that on a two-table query with a selective filter, your engine runs at least an order of magnitude faster with the optimizer enabled than without.

To keep the scope realistic for a semester project, the SQL subset is restricted to queries over \textbf{at most two tables at a time}. The benchmark data set contains three tables in total, and each query joins exactly two of them. Support for queries that reference three or more tables in a single \texttt{FROM} clause is available as a bonus (see Section 9).

\section{Supported SQL Subset}

Your engine must handle the following, and \textbf{nothing else}. Anything outside this subset may be rejected with an error message.

\subsection{Data Definition}
\begin{itemize}
\item \texttt{CREATE TABLE} \textit{name} \texttt{(} \textit{column declarations} \texttt{)} where each column is \texttt{INT}, \texttt{DOUBLE}, or \texttt{VARCHAR}. No constraints. No \texttt{NOT NULL}. No primary keys.
\end{itemize}

\subsection{Query}
\begin{itemize}
\item \texttt{SELECT} followed by a comma-separated list of qualified column references (\texttt{table.col}). No \texttt{*}, no expressions, no aliases, no aggregates.
\item \texttt{FROM} followed by either a single table name or two comma-separated table names. No \texttt{JOIN} keyword; two-table queries always use the comma-separated form. No table aliases. Support for three or more tables is a bonus feature (Section 9).
\item \texttt{WHERE} followed by a conjunction (\texttt{AND}-separated) of simple predicates. Each predicate is one of:
  \begin{itemize}
  \item \texttt{table.col} \textit{op} \texttt{table.col} (a join condition or a column-to-column filter)
  \item \texttt{table.col} \textit{op} \textit{literal} (a filter against a constant)
  \end{itemize}
  where \textit{op} is one of \texttt{=}, \texttt{<}, \texttt{>}, \texttt{<=}, \texttt{>=}, \texttt{!=}.
\item \texttt{LIMIT} \textit{n}.
\end{itemize}

There is no \texttt{GROUP BY}, no \texttt{ORDER BY}, no \texttt{HAVING}, no \texttt{DISTINCT}, no \texttt{NULL}, no \texttt{OR}, no parentheses in the \texttt{WHERE} clause, no nested queries, no aggregate functions. The \texttt{WHERE} clause is always a flat list of predicates joined by \texttt{AND}. This is by design: the smaller the surface area of the language, the more time you can spend on the parts that matter.

\subsection{Diagnostic and Shell Commands}
\begin{itemize}
\item \texttt{EXPLAIN} \textit{query} --- prints the query plan twice, once without the optimizer and once with it, side by side as shown in the example above.
\item \texttt{EXPLAIN ANALYZE} \textit{query} --- runs the query both ways, prints the wall-clock time and row count for each, and reports the speedup ratio.
\item \verb|\d| --- list loaded tables and their row counts.
\item \verb|\load |\textit{table}\verb| |\textit{file.csv} --- load rows from a CSV file into a previously created table.
\item \verb|\q| --- quit.
\end{itemize}

\section{Architectural Overview}

Your system is organized as a pipeline of five stages. Each stage has a single, well-defined job. Keep them strictly separated; do not let parsing logic leak into the optimizer or execution logic leak into the parser.

\begin{codebox}
\begin{lstlisting}
SQL text
  |
  v
[ 1. Lexer ]      ---> token stream
  |
  v
[ 2. Parser ]     ---> query AST
  |
  v
[ 3. Planner ]    ---> logical plan tree (validated against catalog)
  |
  v
[ 4. Optimizer ]  ---> optimized plan tree (predicate pushdown applied)
  |
  v
[ 5. Executor ]   ---> rows
\end{lstlisting}
\end{codebox}

\subsection{The Catalog}
A simple in-memory map from table name to a list of (column name, column type) pairs, plus the row count once data is loaded. Used by the planner to validate column references and by the executor to interpret CSV rows. The catalog does not need to be persisted to disk; it is rebuilt at startup as the user runs \texttt{CREATE TABLE} and \verb|\load|.

\subsection{The Row Representation}
A row is an array of values. A value is one of: a 64-bit integer, a double, or a string. Use a tagged union in C, or a \texttt{struct} with a type field, or a \texttt{std::variant} in C++. The same value type is used throughout the executor and the predicate evaluator.

\section{Phase 1 --- Parser and Plan Generation}

The goal of Phase 1 is to take a SQL string and produce a validated logical plan tree. By the end of this phase, the \texttt{EXPLAIN} command must print a plan tree (the unoptimized version is enough; the optimized version arrives in Phase 2). Malformed queries must be rejected with an error message that mentions the line and column where the problem was found.

\subsection{Lexer}
Write a hand-written lexer. It reads characters from the SQL string and produces a sequence of tokens. The token kinds you need are:

\begin{itemize}
\item \textbf{Keywords}: \texttt{SELECT}, \texttt{FROM}, \texttt{WHERE}, \texttt{AND}, \texttt{LIMIT}, \texttt{CREATE}, \texttt{TABLE}, \texttt{INT}, \texttt{DOUBLE}, \texttt{VARCHAR}. Keywords are case-insensitive.
\item \textbf{Identifiers}: a letter or underscore followed by letters, digits, or underscores. Used for table and column names.
\item \textbf{Integer literals}: a sequence of digits.
\item \textbf{Double literals}: digits with a decimal point.
\item \textbf{String literals}: characters enclosed in single quotes. You do not need to handle escape sequences.
\item \textbf{Operators}: \texttt{=}, \texttt{<}, \texttt{>}, \texttt{<=}, \texttt{>=}, \texttt{!=}.
\item \textbf{Punctuation}: \texttt{,}, \texttt{(}, \texttt{)}, \texttt{.}, \texttt{;}.
\end{itemize}

Whitespace separates tokens and is otherwise discarded. The lexer needs to track the line and column of each token so the parser can produce useful error messages. The lexer is the simplest component in the project; expect it to be a few hundred lines of straightforward code.

\subsection{Parser}
Write a hand-written recursive-descent parser. Do not use a parser generator (Bison, Yacc, ANTLR, or any equivalent). The parser consumes the token stream from the lexer and produces an abstract syntax tree (AST).

You only need to parse two kinds of statements: \texttt{CREATE TABLE} and \texttt{SELECT}. The grammar is small enough that the parser is essentially a translation of the grammar into a function for each rule. A sketch of the grammar in EBNF:

\begin{codebox}
\begin{lstlisting}
statement      ::= create_stmt | select_stmt

create_stmt    ::= 'CREATE' 'TABLE' IDENT '(' col_decl (',' col_decl)* ')'
col_decl       ::= IDENT type
type           ::= 'INT' | 'DOUBLE' | 'VARCHAR'

select_stmt    ::= 'SELECT' col_ref (',' col_ref)*
                   'FROM' IDENT (',' IDENT)?
                   ('WHERE' predicate ('AND' predicate)*)?
                   ('LIMIT' INT_LIT)?

col_ref        ::= IDENT '.' IDENT
predicate      ::= operand op operand
operand        ::= col_ref | literal
literal        ::= INT_LIT | DOUBLE_LIT | STRING_LIT
op             ::= '=' | '<' | '>' | '<=' | '>=' | '!='
\end{lstlisting}
\end{codebox}

The AST node types are correspondingly small: \texttt{CreateTableStmt}, \texttt{SelectStmt}, \texttt{ColumnRef}, \texttt{Literal}, \texttt{Predicate}. Each node carries the source location of its first token for error reporting.

\subsection{Validator and Plan Generator}
After the parser produces an AST, walk the AST and check that it makes sense against the catalog. Specifically:

\begin{itemize}
\item Every table named in \texttt{FROM} must exist in the catalog. An unknown table is an error.
\item Every \texttt{table.col} reference must name a column that exists in that table. An unknown column is an error.
\item Every column referenced in \texttt{SELECT} or \texttt{WHERE} must use a table that appears in the \texttt{FROM} list. Referencing a column from a table not in scope is an error.
\item In each predicate, the two operand types must be comparable. Comparing an integer to a string is an error. An integer compared to a double is allowed (the integer is promoted).
\end{itemize}

If validation passes, translate the AST into a logical plan tree. The logical plan node types you need are:

\begin{itemize}
\item \textbf{Scan}: leaf node, names a single table. Output is every row of that table.
\item \textbf{CrossProduct}: two children. Output is every pairing of one row from the left child with one row from the right child.
\item \textbf{Filter}: one child, holds a list of predicates joined by \texttt{AND}. Output is the rows from the child that satisfy every predicate.
\item \textbf{Project}: one child, holds a list of column references. Output is each input row reshaped to contain only those columns.
\item \textbf{Limit}: one child, holds an integer. Output is the first $n$ rows from the child.
\end{itemize}

The translation is mechanical:

\begin{enumerate}
\item If the \texttt{FROM} clause names one table, build a single \texttt{Scan} for it.
\item If the \texttt{FROM} clause names two tables $t_1$ and $t_2$, build \texttt{CrossProduct(Scan($t_1$), Scan($t_2$))}.
\item If the query has a \texttt{WHERE} clause, wrap the result in a \texttt{Filter} whose predicate list is the conjuncts of the \texttt{WHERE}.
\item Wrap that in a \texttt{Project} for the \texttt{SELECT} list.
\item If the query has a \texttt{LIMIT}, wrap the result in a \texttt{Limit}.
\end{enumerate}

Because the \texttt{FROM} clause has at most two tables, the plan tree contains at most one \texttt{CrossProduct} node. This is an unoptimized plan, and it is intentionally bad for the two-table case. That is fine. Phase 2 will fix it.

\subsection{The EXPLAIN Command}
Implement a function that takes a plan tree and prints it as an indented text tree. Each node prints its type and its key parameters (the table name for a scan, the predicates for a filter, the column list for a project). At this stage, \texttt{EXPLAIN} prints only the unoptimized plan; in Phase 2 you will extend it to print the optimized plan as well.

\section{Phase 2 --- Predicate Pushdown}

Phase 2 has one job: implement predicate pushdown. That is the entire optimizer. There is no statistics collection, no cost model, no join reordering, no search algorithm. You will write one tree-rewriting function and run it on the plan tree produced by Phase 1.

\subsection{What Predicate Pushdown Does}
A query like
\begin{codebox}
\begin{lstlisting}
SELECT users.name, orders.total
FROM users, orders
WHERE users.id = orders.user_id
  AND users.country = 'PK';
\end{lstlisting}
\end{codebox}
produces this plan from Phase 1:
\begin{codebox}
\begin{lstlisting}
Project [users.name, orders.total]
  Filter [users.id = orders.user_id AND users.country = 'PK']
    CrossProduct
      Scan users
      Scan orders
\end{lstlisting}
\end{codebox}
Predicate pushdown rewrites it into this:
\begin{codebox}
\begin{lstlisting}
Project [users.name, orders.total]
  HashJoin [users.id = orders.user_id]
    Filter [users.country = 'PK']
      Scan users
    Scan orders
\end{lstlisting}
\end{codebox}
Two things happened. First, the predicate \texttt{users.country = 'PK'} mentions only columns from \texttt{users}, so it was pushed down to filter the \texttt{users} scan directly, before the cross product. Second, the predicate \texttt{users.id = orders.user\_id} is an equality between a column on the left side of the cross product and a column on the right side, so it was pulled into the cross product, turning the cross product into a hash join.

\subsection{The Algorithm}
You will write a single function that takes a plan tree and returns an optimized plan tree. Because the plan has at most one \texttt{CrossProduct} (two-table queries only), the function does not need to recurse. It walks the tree looking for the pattern \texttt{Filter} directly above \texttt{CrossProduct}. If the pattern is not present, the function returns the plan unchanged. If it is present:

\begin{enumerate}
\item Look at each predicate in the filter individually. For each predicate, determine which of the two tables below the cross product it mentions.
\item Sort the predicates into four buckets:
  \begin{itemize}
  \item \textbf{Left-only}: mentions only columns from the left child's table.
  \item \textbf{Right-only}: mentions only columns from the right child's table.
  \item \textbf{Join condition}: an equality (\texttt{=}) with one side referencing the left table and the other side referencing the right table.
  \item \textbf{Remainder}: anything else (for example, a non-equality comparison across the two tables).
  \end{itemize}
\item Wrap the left \texttt{Scan} in a \texttt{Filter} containing the left-only predicates, if any. Do the same for the right \texttt{Scan} with the right-only predicates.
\item If at least one join condition was found, replace the \texttt{CrossProduct} with a \texttt{HashJoin} carrying that join condition. If there was more than one join condition, keep the first as the \texttt{HashJoin} condition and add the rest to the remainder bucket. If no join condition was found, keep the \texttt{CrossProduct}.
\item If the remainder bucket is empty, remove the original \texttt{Filter} above the join (or cross product) entirely. Otherwise keep the \texttt{Filter} with only the remainder predicates.
\end{enumerate}

That is the entire optimizer. No statistics, no cost model, no search, no recursion. For a two-table query, these five steps produce the optimized plan directly.

\subsection{Why This Is Worth Doing}
The unoptimized plan computes the full cross product of the two tables before applying any filter. For tables of a few thousand and a few tens of thousands of rows, the cross product has tens or hundreds of millions of intermediate rows, which takes many seconds to materialize. The pushed-down version filters the left table down to perhaps a few hundred rows first, then runs a hash join that finishes in well under a second. The whole point of an optimizer is to make this kind of difference happen automatically, and predicate pushdown is the single most important rewrite that databases use to achieve it.

\subsection{Extending EXPLAIN}
Update the \texttt{EXPLAIN} command so that it prints \textit{both} plans for the same query, labeled ``WITHOUT optimizer'' and ``WITH optimizer.'' The user should see the difference at a glance. This side-by-side view is what your demo will show.

\section{Phase 3 --- Execution Engine and Benchmark}

Phase 3 makes the plan trees actually run. You will build a simple executor in which each plan node type has a corresponding function that takes its input rows and returns its output rows. To execute a query, you walk the plan tree bottom-up: each node's executor receives the already-computed outputs of its children, does its job on them, and hands its own output back up.

\subsection{Execution Model}
There are two common ways to execute a query plan. The first is the \textit{pipelined} model, in which rows flow up the tree one at a time and every operator is a small state machine that yields the next row on demand. This is what real databases do, and it is also hard to get right: each operator has to remember where it was between calls, and debugging a bug in the middle of a deep pipeline is unpleasant.

The second way is the \textit{materialized} model. Each operator receives its complete input as a list (or array, or vector) of rows, does its entire job, and returns its complete output as another list of rows. There is no state held between calls because each call does all the work at once. This is the model you will use.

The tradeoff is memory: materialized execution holds the full intermediate results in RAM at every level of the plan. For a small data set this is fine. For the unoptimized version of your benchmark queries, however, the cross product of two tables becomes enormous, and your program will spend many seconds (or run out of memory) trying to materialize it --- which is precisely the point of the benchmark. The unoptimized version is \textit{supposed} to be dramatically slow. Predicate pushdown makes the intermediate results small enough to fit comfortably in memory, which is exactly why the speedup is so large.

\subsection{The Row and Value Types}
A row is an array of values. A value is one of: a 64-bit integer, a double, or a string. Use a tagged union in C, or a \texttt{struct} with a type field, or a \texttt{std::variant} in C++. Comparison rules: integers compare as integers; doubles compare as doubles; strings compare lexicographically; an integer compared to a double promotes the integer. Comparing values of incompatible types is a runtime error and should abort the query with a clear message.

A table is a list of rows. In C++, use \texttt{std::vector<Row>}. In C, a dynamic array that grows as rows are appended works equally well.

\subsection{The Executor Functions}
Write one function per plan node type. Each function takes whatever inputs it needs and returns a list of rows. There are six of them, and none of them is longer than a screen of code.

\subsubsection{Scan}
\begin{codebox}
\begin{lstlisting}
vector<Row> exec_scan(string table_name);
\end{lstlisting}
\end{codebox}
Open the CSV file for the named table, read each line, parse it into a \texttt{Row} according to the column types from the catalog, and append it to a result vector. When the file ends, return the vector. That is the whole function.

\subsubsection{Filter}
\begin{codebox}
\begin{lstlisting}
vector<Row> exec_filter(vector<Row> input, vector<Predicate> preds);
\end{lstlisting}
\end{codebox}
Allocate an empty result vector. For each row in the input, evaluate every predicate against the row using the predicate evaluator (described below). If every predicate passes, append the row to the result. Return the result.

\subsubsection{Project}
\begin{codebox}
\begin{lstlisting}
vector<Row> exec_project(vector<Row> input, vector<ColumnRef> cols);
\end{lstlisting}
\end{codebox}
For each input row, construct a new row containing only the values at the requested column positions, in the requested order. Return the list of new rows.

\subsubsection{CrossProduct}
\begin{codebox}
\begin{lstlisting}
vector<Row> exec_cross_product(vector<Row> left, vector<Row> right);
\end{lstlisting}
\end{codebox}
Two nested loops. For each row $\ell$ in \texttt{left}, for each row $r$ in \texttt{right}, concatenate $\ell$ and $r$ into a combined row and append it to the result. Return the result. This is where the unoptimized plan spends all its time.

\subsubsection{HashJoin}
\begin{codebox}
\begin{lstlisting}
vector<Row> exec_hash_join(vector<Row> left, vector<Row> right,
                           JoinCondition cond);
\end{lstlisting}
\end{codebox}
Build a hash map from the left side: for each row in \texttt{left}, compute the value of the left join column, and insert the row into the map under that key. (Multiple rows may share the same key; use a map from key to a list of rows, or a multimap.) Then, for each row in \texttt{right}, look up the value of the right join column in the map; for every matching left row, concatenate and append to the result. Return the result.

\subsubsection{Limit}
\begin{codebox}
\begin{lstlisting}
vector<Row> exec_limit(vector<Row> input, int n);
\end{lstlisting}
\end{codebox}
Return a copy of the first \texttt{min(n, input.size())} rows of \texttt{input}. That is all.

\subsection{The Execute Dispatcher}
Glue the executor functions together with a single recursive function that takes a plan node and returns its output rows. In C++ this looks like:

\begin{codebox}
\begin{lstlisting}
vector<Row> execute(PlanNode* node) {
    switch (node->type) {
        case NODE_SCAN:
            return exec_scan(node->scan.table_name);
        case NODE_FILTER:
            return exec_filter(execute(node->filter.child),
                               node->filter.preds);
        case NODE_PROJECT:
            return exec_project(execute(node->project.child),
                                node->project.cols);
        case NODE_CROSS_PRODUCT:
            return exec_cross_product(execute(node->cross.left),
                                      execute(node->cross.right));
        case NODE_HASH_JOIN:
            return exec_hash_join(execute(node->join.left),
                                  execute(node->join.right),
                                  node->join.cond);
        case NODE_LIMIT:
            return exec_limit(execute(node->limit.child),
                              node->limit.n);
    }
}
\end{lstlisting}
\end{codebox}

Running a query end to end is then one call: \texttt{execute(optimized\_plan)}. To measure \texttt{EXPLAIN ANALYZE} timings, wrap the \texttt{execute} call in a clock-reading pair.

\subsection{The Predicate Evaluator}
The filter and hash join functions both need to evaluate predicates. Write one function that takes a predicate (an AST node) and a row (or a pair of rows, for a join predicate), looks up the column values from the row, applies the operator, and returns true or false. This is a short function whose body is a switch statement over the six operators (\texttt{=}, \texttt{<}, \texttt{>}, \texttt{<=}, \texttt{>=}, \texttt{!=}). Use the comparison rules given in Section 7.2.

\subsection{The SQL Shell}
Build a REPL that reads a statement from standard input (terminated by a semicolon), runs it through the full pipeline, and prints the result. The shell supports:

\begin{itemize}
\item Multi-line input until a terminating semicolon.
\item Tabular output with column headers and aligned columns.
\item A row count and elapsed time printed after each query.
\item The backslash commands listed in Section 3.3.
\item \texttt{EXPLAIN} and \texttt{EXPLAIN ANALYZE}, both of which produce side-by-side output as shown in the example session.
\end{itemize}

For \texttt{EXPLAIN ANALYZE}, run the query twice: once by passing the unoptimized plan to \texttt{execute}, and once by passing the optimized plan to \texttt{execute}. Time each call with a clock. Print both times and the ratio.

\subsection{Benchmark}
The final deliverable is a benchmark that demonstrates your optimizer is doing real work. You will load a fixed data set consisting of three tables. The tables are sized so that unoptimized two-table queries are slow but finite --- a few seconds to perhaps a minute --- and optimized queries finish almost instantly. You do not need to worry about running out of memory on the unoptimized runs; the provided data has been sized with this executor in mind.

You will run three queries against this data set. Each query joins exactly two of the three tables and has at least one predicate that is a column-to-literal comparison. The choice of which two tables each query uses is yours, as long as all three tables appear in at least one query across the suite.

For each query, your benchmark must print:
\begin{itemize}
\item The plan tree without the optimizer.
\item The plan tree with the optimizer.
\item The wall-clock execution time without the optimizer.
\item The wall-clock execution time with the optimizer.
\item The speedup ratio.
\end{itemize}

A successful submission shows speedups of at least \textbf{10x} on at least two of the three queries. The optimized version should be measurably and visibly faster, not by a few percent but by a large factor.

\section{Bonus Opportunities}

Each of the following is independent and may be attempted in any combination. Bonus credit is awarded on top of the base grade up to a maximum of 15 percentage points total. To claim bonus credit, your design document must include a dedicated section explaining which bonus items you attempted and how you verified them.

\subsection{Three-Table Queries (up to 10\%)}
Extend the parser, planner, optimizer, and executor to support queries that reference three tables in the \texttt{FROM} clause. The grammar change is minor: allow \texttt{FROM} to list any number of tables. The planner change builds nested \texttt{CrossProduct} nodes left-to-right: \texttt{CrossProduct(CrossProduct(Scan($t_1$), Scan($t_2$)), Scan($t_3$))}.

The optimizer change is where the real work is. Your pushdown function must now recurse into the children of the plan tree after processing the outer filter, so that predicates can also be pushed into the inner cross product. A predicate that mentions only $t_3$ needs to travel through two levels of cross product to land on \texttt{Scan($t_3$)}. A predicate that mentions $t_1$ and $t_3$ is an equi-join condition for the outer cross product rather than the inner one. Thinking carefully about which predicates belong at which level is the point of this bonus.

The benchmark must include at least one three-table query and show its speedup, in addition to the required two-table queries.

\subsection{Sort-Based ORDER BY (up to 5\%)}
Add \texttt{ORDER BY} support to the grammar, the planner, and the executor. The new operator sorts its input fully in memory (an in-memory sort is sufficient; no external sort required) before yielding rows. The benchmark must include at least one query using \texttt{ORDER BY} and a screenshot or printed output showing correctly ordered results.
